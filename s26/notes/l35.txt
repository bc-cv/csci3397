**[7 min] 3. Phase 1 Recap: You Already Built This (steps 00–06)**

**[22 min] 4. HOW: Phase 2 — Event-Driven Refactor (steps 07–10)**
- The big idea: a pub/sub event bus sits between "where messages come from" and "who processes them."

**[6 min] (a) Event Bus (step 07)**
- Problem: CLI-coupled agent can't serve Telegram + Discord + cron at once.
- Solution: asyncio queue + subscribers.
- Two event types: `InboundEvent` (message in), `OutboundEvent` (response out).
- `AgentWorker` subscribes InboundEvent → runs session → publishes OutboundEvent.
- Step 07 has **no user-visible change** — pure refactor. Keystone for phases 3–4.
- **Punchline**: Decoupling looks pointless until you need 5 I/O channels; then it's everything.

**[3 min] (b) Config Hot Reload (step 08)**
- Watchdog watches workspace; `Config.reload()` re-merges on file change.
- Two-layer config: `config.user.yaml` (human-edited) + `config.runtime.yaml` (process-written).
- Deep-merge so runtime overrides user without clobbering.
- **Punchline**: Long-running servers can't restart on every edit.

**[8 min] (c) Channels (step 09)**
- `EventSource` ABC: platform-specific id (e.g., `platform-telegram:chat:user`).
- `Channel` ABC: `run`/`reply`/`stop` per platform.
- `ChannelWorker` (inbound) publishes to bus; `DeliveryWorker` (outbound) routes to channel's `reply()`.
- Source→session cached in `config.runtime.yaml` (each chat maps to one session).
- **Reliability**: write OutboundEvent to disk before dispatch; `ack()` deletes on successful delivery; recovery replays pending events on startup.
- **Punchline**: Same agent, now talking to your phone. Messages don't get lost across crashes.

**[5 min] (d) WebSocket (step 10)**
- FastAPI `/ws` endpoint — another event sink alongside channels.
- `WebSocketWorker` subscribes both InboundEvent and OutboundEvent, broadcasts JSON to connected clients.
- Enables custom UIs, browser clients, programmatic integrations.
- **Punchline**: Event bus = N sources in, N sinks out, 1 agent runtime.

**[6 min] 5. Wrap-Up: What Phase 2 Buys You**

| | nano_cc (L33) | OpenClaw Phase 1+2 |
|---|---------------|--------------------|
| Entry points | CLI only | CLI + Telegram + Discord + WebSocket |
| Messages | in-memory | persisted, resumable, recoverable |
| Skills | all loaded | lazy via `SKILL.md` |
| Context | accumulate forever | compact at 80%, rollover |
| Config | restart to change | hot reload |

- **3 takeaways**:
  1. Phase 1 (skills, persistence, compaction) makes a single agent survive long sessions.
  2. Event bus decouples I/O from logic — the one refactor that enables everything downstream.
  3. Channels + WebSocket = multi-platform for free once the bus exists.
- **Next L36**: Phase 3 autonomy (cron, dispatch, memory) + Phase 4 production (concurrency). Agent works while you sleep.
